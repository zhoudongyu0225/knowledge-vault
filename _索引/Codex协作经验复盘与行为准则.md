## 一、执行经验总结

| 问题做法 | 正确执行方式 | 教训 |
|---|---|---|
| 在需求未定清时直接实现，容易做偏。 | 先把多种理解列出来，定义完成标准，再动手。复杂任务先给步骤清单，每一步说明如何验收。 | 用户更重视“先想清楚再动手”，速度不能替代方向正确。 |
| 写文档时只写定位、卖点、抽象判断，缺少数据。 | 核心业务档案必须包含数据快照、来源、口径、趋势、决策含义和待验证项。 | “虚”的文档无法支持决策。尤其 ROK/COD 这类产品档案必须数据驱动。 |
| 看高分辨率截图时只凭整体观感下结论。 | 优先读取本地原图，必要时裁剪、放大、分区识别；看不清就标注不确定，不得假装看清。 | 对核心业务数据不能用模糊印象做判断。 |
| 用户给敏感内部数据或付费群信息时，直接大段保存原文。 | 默认 `visibility: private`；只沉淀摘要、判断、数据口径、行动项，不保存不必要的原文和个人信息。 | 知识库要可用，也要控制隐私、版权和泄露风险。 |
| 用户暴露 GitHub token 后继续按普通文本处理。 | 立即视为泄露凭据，提醒撤销；不得写入文档、Git、日志摘要或最终回复。后续用凭据管理器或 `gh auth login`。 | secret 一旦出现在对话中，就应按已泄露处理。 |
| 假设 GitHub 推送一定可行。 | 先确认本地 remote、分支、认证状态；能本地 commit 就本地提交，push 失败要明确说明原因。 | 不要把“本地已提交”和“远端已同步”混为一谈。 |
| 把 AGENTS、知识文档、skill 混在一起。 | 长期稳定原则放 AGENTS；项目事实和方法论放知识库；重复且稳定的工作流才做成 skill。 | 不同载体承担不同职责，避免一个文件变成资料堆。 |
| 为了“以后可能会用”过早抽象或做复杂系统。 | 先解决真实高频问题，再沉淀流程；只有重复、边界清楚、能省解释的任务才 skill 化。 | 用户偏好简单有效，不接受为了未来可能性堆复杂度。 |
| PowerShell 命令写得很长，管道和字符串拼接复杂，导致解析错误或超时。 | 用 `rg`、`Get-Content -Encoding UTF8`、小步查询；长逻辑用清晰脚本；避免超大递归和无界输出。 | Windows 下要降低命令复杂度，先取样再扩展。 |
| 读取中文文件时未指定编码，造成乱码。 | PowerShell 读取 Markdown、JSON、配置文件时优先加 `-Encoding UTF8`。 | 编码问题会直接影响判断质量。 |
| 对网页、公众号、规则、价格、工具状态等易变化信息凭记忆回答。 | 需要最新信息、链接、引用或高风险决策时必须浏览或读取官方/原始来源。 | 时间敏感信息要核查，不要凭旧知识。 |
| 创意脚本一次性给最终稿。 | 先共创：项目校准、方向卡、大纲、用户确认，再写详细脚本或导演简报。 | 用户更喜欢讨论式打磨，尤其买量创意不是一次性生成。 |
| 做 QA 或开发时没有独立验收。 | 修改后运行最小验证：`git status --short`、`git diff --check`、相关测试或人工检查；最终报告验证结果。 | 完成不是“我改了”，而是“验证条件成立”。 |
| 遇到工具失败后继续堆命令。 | 先归因：权限、路径、编码、依赖、网络、文件锁、超时。换更简单的验证路径。 | 失败后要收敛，不要扩大混乱。 |

## 二、用户偏好与理念档案

## 1. 工作方式偏好

- 偏好直接、务实、能落地的协作，不喜欢空泛赞同和大段虚话。
- 不确定时宁愿先问，也不要自作主张。
- 一个问题有多种理解时，要先列出理解差异。
- 复杂任务要先定义目标和验收标准，再执行。
- 发现更简单的办法要主动说，不要为了显得专业而复杂化。
- 只改和当前任务直接相关的内容，不顺手重构或扩大范围。

## 2. 决策偏好

- 重视数据、事实、来源、证据强度和可验证假设。
- 对游戏发行、买量、广告创意、产品判断有深经验，AI 不应替代最终判断。
- AI 的价值是预处理信息、结构化分析、发现异常、拆解假设、沉淀方法，帮助人更快决策。
- 面对冲突判断时，要列出冲突点、证据、假设和验证方式。
- 分析终点必须是下一步动作，而不是只解释现象。

## 3. 知识库偏好

- 知识库要服务调用，不是收藏夹。
- 用户可以直接把有价值内容发给 Codex，由 Codex 负责整理、归档和关联。
- 查找时用户不需要记路径，应能用自然语言按问题调用。
- 知识库要按月沉淀和优化；高价值信息进文件，重复工作流沉淀成 skill。
- 中文命名可以使用，不应无理由强行英文。
- 工作主轴是游戏发行和广告，其它领域、个人收藏要分开标记。

## 4. 游戏发行与广告偏好

- 核心领域：海外游戏发行、买量广告、广告创意、投放优化、数据分析、竞品监控、市场洞察、产品反哺。
- ROK/COD 等产品分析必须结合生命周期数据、渠道结构、自然 vs 广告、iOS vs AOS、付费率、ARPU、ARPPU、留存和回收。
- SLG 买量创意要轻用户可理解，前三秒能看懂玩法或冲突。
- 当前 SLG 广告更重视 X+SLG 的入口玩法、可见反馈和清晰循环，不能过重讲联盟规则、科技树和大地图规则。
- COD 不能写成 ROK 换皮；要突出龙、英雄、怪物压力、野外探索、奇幻生态和即时反馈。
- ROK 当前更偏成熟长线存量经营，要重视回流、老用户价值和高质量新增。

## 5. 安全与隐私偏好

- 内部数据、付费群、聊天记录、截图默认私有。
- 不把敏感资料外发，不把 secret 写入知识库或 Git。
- 引用付费群内容时保留观点和判断，不保留可识别个人或大段原文。
- 对外部来源、公众号、网页要尊重版权，优先摘要和提炼。

## 三、可复用规则清单

## 启动任务

- 先判断任务类型：问答、整理、写文档、开发、调研、创意、数据分析、知识库维护。
- 明确完成标准：做完什么后，用什么方式验证成立。
- 不确定时先问；可合理假设时要明示假设。
- 复杂任务先列步骤，并随着执行更新状态。

## 读取与检索

- 搜索文件优先用 `rg` 或 `rg --files`。
- 中文文件优先 `Get-Content -Encoding UTF8`。
- 大目录、大日志先枚举和抽样，再定向检索，不无界全量输出。
- 读取图片和截图时优先用原始本地文件；看不清就裁剪、放大或标注不确定。

## 写入知识库

- 每条正式入库内容要包含：来源、日期、摘要、核心观点、决策用途、证据强度、待验证项、关联资料。
- 业务档案必须有数据，不得只有定位、卖点和抽象判断。
- 重复主题要合并，保留最新、最准确、最可执行版本。
- 敏感内容默认 `visibility: private`。
- 知识库修改后检查 `git status --short` 和 `git diff --check`，必要时本地 commit。

## 数据与决策分析

- 先定义口径，再下结论。
- 关键结论必须尽量附带数据、事实、样本或验证路径。
- 对假设、推断、已验证事实分开写。
- 凡是建议“测试、优化、关注、提升”，必须说明测什么、为什么、怎么测、看什么指标、如何决策。
- 对 ROK/COD 决策，先调用产品档案和历史数据，再做素材、投放或产品判断。

## 创意与脚本

- 不一次性强行给最终稿，先共创。
- 先做项目气质校准，再给方向卡，再根据用户选择扩写。
- 创意必须写清玩法、流程、反馈、状态变化、扩张期待和制作风险。
- 不抄参考素材题材，只提取交互结构、观看结构、压力和反馈。

## 开发与文件修改

- 开始前检查工作区状态。
- 只改当前任务相关文件。
- 不回退用户或其他 agent 的改动。
- 手动编辑用 `apply_patch`。
- 不用破坏性命令，除非用户明确要求。
- 修改后跑最小验证，并在最终回复说明验证结果。

## Git 与远程同步

- 本地 commit 前检查 diff scope。
- commit 信息要描述真实变化。
- push 前确认凭据可用；没有凭据时只本地提交并说明。
- secret 不得进入 Git。发现 token、key、cookie 时按泄露处理。

## Skill 沉淀

- 只有重复、高价值、边界清楚、输入输出稳定的工作流才做成 skill。
- skill 应写清触发场景、输入、步骤、输出格式、验证方式。
- 不把一次性任务或尚未稳定的判断提前 skill 化。

## 最终回复

- 简洁说明做了什么、文件在哪里、验证结果是什么。
- 对没做成的事情明确说明原因和下一步。
- 不用空泛赞美，不用过度道歉。
- 用户看不到工具输出时，要转述关键结果。
