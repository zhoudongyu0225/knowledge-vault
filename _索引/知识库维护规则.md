# 知识库维护规则

## 目标

知识库服务于共同工作，不追求全量收藏。它的价值是把高频、高价值、可复用的信息变成以后能调用的判断、方法和证据。

## 入库原则

入库前先判断三个问题：

1. 这条信息以后会不会再次用到？
2. 它能不能帮助决策、复盘、创意、投放、产品判断或学习？
3. 它是否能被整理成清晰观点、数据、案例、方法或待验证假设？

只有“值得以后调用”的内容才进入正式目录。临时、不确定、待处理的内容先放 `00_收件箱`。

## 整理格式

正式沉淀的内容至少包含：

- 来源和日期。
- 原始信息摘要。
- 核心观点或事实。
- 对当前工作的启发。
- 可用于哪些决策。
- 证据强度。
- 待验证项。
- 关联资料。

涉及敏感数据、付费群、内部资料时：

- 默认 `visibility: private`。
- 不保留不必要的个人信息。
- 不大段搬运付费内容原文，优先转写成自己的摘要、判断和行动项。
- 重要业务结论必须标注“已验证 / 待验证 / 仅假设”。

## 月度沉淀

每月做一次轻量复盘，不做复杂维护。

完成标准：

1. 清理 `00_收件箱` 中值得入库的内容。
2. 合并重复主题，保留最新、最准确、最可执行版本。
3. 更新重要索引和关联链接。
4. 提炼当月新增的高价值判断、方法、信号和待验证假设。
5. 判断是否有重复工作流值得沉淀为 skill。

建议输出：

- 当月新增高价值资料。
- 重要判断更新。
- 被新证据推翻或需要降级的旧判断。
- 可复用工作流候选。
- 下月值得持续观察的问题。

## 工作流沉淀为 Skill 的标准

只有重复出现、边界清楚、能提高效率的工作流才沉淀为 skill。

适合沉淀：

- 微信群观点整理。
- 外部文章总结入库。
- 游戏产品数据档案更新。
- SLG 买量素材拆解。
- 投放复盘模板化。
- 竞品监控和市场信号整理。

暂不沉淀：

- 一次性问题。
- 判断标准还不稳定的工作。
- 需要大量临场业务取舍的任务。
- 只是为了“以后可能会用”的流程。

## 每次修改知识库后的验证

修改知识库后至少检查：

- 文件是否放在合理目录。
- 是否有来源、日期、标签和可见性。
- 是否写清楚“为什么保存”和“以后怎么用”。
- 是否和已有资料重复。
- Git 工作区是否清楚，必要时提交本地版本。
