# 快手弹幕小玩法：全流程开发与发布指南

## 使用说明

本文把快手公开文档整理成“商务准入 → 后台入驻 → PC 客户端与服务端 → 自测验收 → 灰度发布 → 结算运维”的完整链路。

- 核对日期：2026-07-16
- 已核实来源：`open.kuaishou.com` 官方正文
- 未纳入依据：官方页面跳转到 `docs.qingque.cn`、GitHub、下载站等外部正文
- 资料归属：个人弹幕小玩法项目，与本职工作资料分开
- 链接说明：准入概览统一引用官方无后缀 canonical 地址，避免同一页面以多种链接形式重复入库

## 核心结论

1. 快手基础弹幕玩法的主链路是“PC 直播伴侣启动独立 EXE + roomCode + 开发者服务端绑定 + 快手 HTTP 回调”，不能按普通小程序或普通小游戏理解。
2. 后台注册不是第一步：当前入驻页要求先完成商务提案和 PRD 审核，再做主体与项目创建。
3. 事件同样不保证顺序，必须用 `unique_message_id` 幂等；礼物还应使用 `uniqueNo` 做交易级幂等。
4. PC 包体、`ks_manifest.json`、IPC、自动捕获、进程清理和禁止热更新，都是发布链路的一部分。
5. 快手公开页存在资质、性能和能力代际冲突。较新页面只作为当前公开参考，不能仅按更新时间覆盖不同类目、能力或合作阶段的口径；最终由当期商务、后台和联调结果确认。

## 一、完整生命周期

| 阶段 | 前置条件 | 关键动作 | 完成产物 | 放行条件 |
|---|---|---|---|---|
| 1. 判断玩法类型 | 有玩法概念 | 区分“互动小玩法”与“游戏小玩法”，评估是否含内购 | 类型与资质判断 | 资质路径明确 |
| 2. 玩法与合规预审 | 互动规则、初版美术 | 证明不是普通游戏移植；梳理免费/付费反馈、一局闭环、安全区 | 一页玩法说明、红线清单 | 直播互动价值成立 |
| 3. 商务提案与 PRD | 公司、联系人、玩法材料 | 与平台商务建立联系，完成商务提案和 PRD 审核 | 商务准入结论 | 获准进入后台入驻 |
| 4. 主体注册与审核 | 企业资料、管理员快手号 | 邮箱注册、公司资料、管理员扫码与主体提交 | 开发者主体 | 当前公开页约 3 个工作日，仅作动态快照 |
| 5. 创建玩法 | 主体通过 | 创建小玩法，完成后台提案评估 | AppID、AppSecret、项目 | 项目状态允许补全信息 |
| 6. 基础信息与物料 | 名称、图标、规则、教程 | 填写简介、介绍图、开播教程、专栏封面等 | 正式玩法档案 | 信息审核通过 |
| 7. 技术方案 | AppID、scope、回调地址 | 设计 EXE、伴侣、IPC、开发者服务端、快手服务端的链路 | 技术方案、接口与配置表 | scope 与环境准备就绪 |
| 8. PC 客户端开发 | 引擎工程、直播伴侣 | 解析启动参数、制作 manifest、接 IPC、捕获与退出 | 可启动和捕获的玩法包 | AppID.exe、进程与画面行为正确 |
| 9. 服务端与回调 | 公网 HTTPS 服务 | token、bind/round/gift、验签、幂等、乱序、限流和监控 | 稳定事件入口 | 回调正确返回并通过异常测试 |
| 10. 自测与私密联调 | 已上传自测包 | 配置体验者，覆盖完整包，验证评论/点赞/礼物/退出 | 录屏、问题单、自测记录 | 正常与异常链路通过 |
| 11. 性能、安全与报告 | 功能基本稳定 | 功能、客户端性能、服务端压测、安全检查 | 三类测试报告与录屏 | 采用平台确认的当前验收基线 |
| 12. 包体提审与发布 | 信息、自测、报告完成 | 上传正式 EXE 包，提交审核材料 | 审核通过版本 | 包体合法、安全、稳定、体验合格 |
| 13. 灰度与线上运维 | 监控、回滚、on-call 就绪 | 按平台安排灰度、巡检、告警、止损 | 线上版本、巡检记录 | 指标稳定后逐步放量 |
| 14. 结算、迭代与下线 | 有营收或版本变化 | 账单、发票、重新提审；下线处理权益与数据 | 结算/版本/下线记录 | 不热更新绕审，用户权益闭环 |

## 二、准入、资质与提案

### 2.1 最新公开资质口径

2026-04-22 更新的企业服务类目页，是当前公开资料中时间较新的口径：

| 类型 | 公开资质要点 | 项目判断 |
|---|---|---|
| 社交/直播/互动小玩法 | 营业执照 | 偏简单互动、工具或玩法 |
| 社交/直播/游戏小玩法 | 营业执照、软件著作权，并按是否内购涉及版号或小游戏作品备案 | 更像完整游戏、存在成长/内购时重点确认 |

- 含内购的游戏小玩法涉及版号要求。
- 无内购的游戏小玩法需按备案指引准备监管材料；已有版号时按当前备注处理。
- 这并不自动推翻后台或商务的项目级要求。

官方依据：[2026 企业主体服务类目与资质](https://open.kuaishou.com/docs/operate/Hierarchy/service-category/IndustryCategoryClassification.html)。

### 2.2 官方页面之间的资质冲突

- 2023 准入概览写过注册资本大于 100 万元、软著/电子版权证书等门槛。
- 2024 后台入驻页把软件著作权栏标为选填。
- 2026 类目页按“互动小玩法/游戏小玩法”重新区分资质，不再展示注册资金门槛。

当前处理：以 2026 类目页作为最新公开参考，但在提案前把“主体类型、注册资金、软著、版号/备案”四项发给商务书面确认。官方联系页当前提供 `kszb-wfzm@kuaishou.com`。

### 2.3 提案审查重点

- 玩法必须围绕直播当局互动，不能只把现有游戏接入评论和礼物。
- 禁止抄袭、简单复刻，重点评估互动价值、内容属性、美术品质和商业空间。
- 主播真人画面、评论区、指令和规则要在竖屏直播内可读。
- 要说清楚一局开始、结束、结果与再来一局。
- 成长和道具宜在当局内生效并及时清空，避免形成普通游戏式长期系统。
- 当前准入页对棋牌、转盘、捕鱼、抽奖返现、复杂商店/二级货币、付费唯一决定排行榜、挂机成长等方向风险较高或明确限制。

官方依据：[玩法准入指引](https://open.kuaishou.com/miniPlayDocs/introduction/Barragegameplay/guide.html)、[设计原则](https://open.kuaishou.com/miniPlayDocs/plan/principle.html)。

## 三、画面与交互设计

旧介绍页给出的基础形态是竖屏、视频单直播间、真人 + 玩法画面、观众评论/点赞/送礼参与，并明确禁止挂机开播。2024 技术总览又出现双人连屏、云开播、组队语音、可操作玩法等扩展能力，因此不能把旧版“仅单直播间”当成今天的全部能力边界。

按 1080 宽设计时，当前设计指南给出：

- 顶部约 340 px、底部评论区域约 620 px 应避让，中部玩法主区域高度约 960 px。
- 关键 UI 控件安全区约为 880 × 960。
- 横版规则挂件宽不超过 748 px、高 124 px。
- 竖版规则挂件宽 240 px、高不超过 424 px。
- 规则、指令、付费效果和结果要清晰可读，不能被主播画面或评论遮挡。

这些是设计基准，不是简单把 1080×1920 画布裁成固定像素；不同能力和伴侣布局仍需真机检查。[弹幕玩法设计指南](https://open.kuaishou.com/miniPlayDocs/plan/standard)

## 四、技术架构

~~~mermaid
sequenceDiagram
    participant K as 快手 PC 直播伴侣
    participant G as 开发者 EXE
    participant S as 开发者服务端
    participant O as 快手开放平台

    K->>G: 启动：-c roomCode / -pos / -ipc
    K-->>G: 必要时 IPC 补发 SC_SET_CODE
    G->>S: 上报当前 roomCode 与客户端会话
    S->>O: 获取 access_token
    S->>O: bind(start) 绑定房间
    O-->>S: LIVE_INTERACTION_DATA 回调
    S->>S: 验签、测试隔离、幂等、乱序处理
    S-->>O: HTTP 200 + result=1
    S-->>G: 标准化玩法事件
    G->>G: 渲染反馈
    K-->>G: SC_QUIT / OnDisconnect
    S->>O: bind(stop)
~~~

### 4.1 鉴权

- `getAccessToken` 使用 `app_id + app_secret + client_credentials`。
- 当前 token 有效期写为 48 小时，新旧 token 在旧值未过期时可同时有效。
- 弹幕玩法页建议每小时刷新一次，用来提前发现授权异常；获取失败要重试并告警。
- OpenAPI 的 `app_id` 和 `access_token` 必须放在 query，不能只写进 JSON body。
- 正式联调前需确认 AppID 已获得对应 scope。

官方依据：[接口调用凭证](https://open.kuaishou.com/docs/develop/server/getAccessToken.html)、[玩法技术基础能力](https://open.kuaishou.com/miniPlayDocs/technologyrelated/Affiliatedability/basicability/localdevelopment.html)。

### 4.2 房间绑定、对局与礼物

| 能力 | 用途 | 关键规则 |
|---|---|---|
| `/smallPlay/bind` | `start / stop / status` 房间绑定 | 客户端启动后尽快 bind；当前单 AppID 10 次/秒 |
| `/smallPlay/round` | 对局开始/结束 | 同局 roundId 不变，同一 roomCode 各局不可重复；无对局玩法可不接 |
| `/smallPlay/gift` | 置顶礼物及效果说明 | 当前单 AppID 10 次/秒；礼物与说明需和平台配置一致 |

有对局时要把 `round_id` 贯穿客户端、服务端、事件和日志；否则迟到消息很容易污染下一局。

### 4.3 回调、验签和响应

- 平台 POST 到登记的 callback。
- 使用原始 HTTP body 拼接 AppSecret 后计算 MD5，与 `kwaisign` 比较；不能先反序列化再重组正文。
- 正确消费后返回 HTTP 200 且 JSON `result=1`；否则平台会按配置重试。
- `unique_message_id` 用于整条消息幂等；礼物的 `uniqueNo` 用于交易级幂等。
- 平台不保证评论、点赞、礼物按观众操作顺序到达。
- 正式事件为 `LIVE_INTERACTION_DATA`，压测事件为 `LIVE_INTERACTION_DATA_TEST`。
- 压测消息 ID 和用户 ID 带 `stress_` 前缀，必须从正式榜单、收入、用户资产和业务上报中隔离。

官方依据：[签名与回调附录](https://open.kuaishou.com/docs/develop/server/epay/appendix.html)、[玩法技术基础能力](https://open.kuaishou.com/miniPlayDocs/technologyrelated/Affiliatedability/basicability/localdevelopment.html)。

### 4.4 当前公开事件

| 内部事件 | 快手 `pushType` | 注意事项 |
|---|---|---|
| GIFT | `giftSend` | 只推已订阅礼物；`uniqueNo` 交易级幂等 |
| COMMENT | `liveComment` | 预先配置且精准匹配关键词 |
| LIKE | `liveLike` | 有每人频控和防刷；适合聚合处理 |
| FOLLOW | `follow` | 以当前 AppID 权限为准 |
| SHARE | `share` | 以当前 AppID 权限为准 |
| FANS_JOIN | `fansGroupJoin` | 加入粉丝团 |
| NOBLE | `noble` | 开通/续费贵族，商业规则单独确认 |

超粉团示例把 `pushType` 写成 `liveLike`，与语义明显冲突，不能靠猜测修正；开发前用平台联调消息确认。超粉团、贵族等事件是否参与分成也应按当前后台/协议处理。[技术接口总览](https://open.kuaishou.com/miniPlayDocs/technologyrelated/Affiliatedability/expand/Subsidiaryability)

## 五、PC 客户端、Manifest 与 IPC

### 5.1 启动参数

- `-c roomCode`：当前直播间短码。
- `-pos x,y,width,height`：平台建议的窗口位置和大小。
- `-ipc name`：本地 IPC 名称；只有启用对应 IPC 能力时才会传。
- 如果玩法早于开播启动，伴侣会在开播后通过 `SC_SET_CODE` 补发 roomCode。
- 即使命令行已拿到 roomCode，再次收到 `SC_SET_CODE` 时也必须更新为新值。
- 收到 `SC_QUIT` 或 IPC 断连时，应退出玩法、停止绑定并清理子进程。

### 5.2 包体根目录

根目录需要 UTF-8 编码的 `ks_manifest.json`，至少核对：

| 字段 | 用途 |
|---|---|
| `launcherFileName` | 根目录启动程序，必须命名为 `AppID.exe` |
| `workFileNames` | 伴侣监控的实际工作进程 |
| `killFileNames` | 退出时要清理的进程 |
| `captureParam` | 捕获类型、窗口名、窗口类和进程名 |
| `ipcSetCode` | 是否接收 `SC_SET_CODE` |
| `ipcQuit` | 是否接收 `SC_QUIT` |
| `material` | 横版/竖版规则素材 |

- Unity/UE 当前支持 `game` 与 `window` 捕获。
- Cocos 当前只支持 `window` 捕获。
- 相机模式还需配置虚拟摄像头 deviceId 与分辨率。
- 官方明确禁止游戏包热更新；任何包体变化都应重新走后台审核。

文档存在 `launcherFileName/launchFileName`、`captureParam/capturnParam` 混用，实施时以正式 manifest 说明、伴侣实际解析和联调结果为准。[PC 客户端接入](https://open.kuaishou.com/miniPlayDocs/technologyrelated/Affiliatedability/basicability/clientside)

## 六、自测、测试报告与验收

### 6.1 自测

- 上传自测包后，在项目成员中增加体验者；当前页面写成员总数不超过 90。
- 当前页面同时要求“使用最新 PC 直播伴侣”和最低版本 `5.9.0.1851`，实际以平台当期要求为准。
- 本地覆盖包时按 AppID 目录替换完整包，避免只换 EXE 导致 manifest 或资源不一致。
- 私密直播至少完成：开播、关键词命中/不命中、点赞、每种礼物、再来一局、正常关闭、异常退出。
- 榜单不能只按付费维度决定，并要验证单场直播/单局结算边界。
- 提交 PC 开播、评论、点赞和送礼录屏，保留服务端日志与客户端版本哈希。

### 6.2 报告要求

| 变更类型 | 功能报告 | 客户端性能 | 服务端性能 |
|---|---|---|---|
| 首次提交 | 必填 | 必填 | 必填 |
| 重大迭代 | 必填 | 必填 | 必填 |
| 轻微迭代 | 必填 | 建议/按当期要求 | 建议/按当期要求 |

服务端混合场景压测、监控图和异常说明是重要证据；测试模板是红蓝对战示例，非对战玩法应按自己的机制改写，不能机械照抄“兵种/血量”等用词。[CP 测试报告](https://open.kuaishou.com/miniPlayDocs/technologyrelated/cp/testingreport)

### 6.3 性能标准存在冲突

| 指标 | 2023 测试报告页 | 2024 技术总览页 |
|---|---|---|
| 客户端 FPS | ≥26 | >18 |
| 卡顿/卡帧 | 卡顿率 ≤2% | 卡帧率 <30% |
| CPU | ≤60% | 未在同一口径给出 |
| 稳定性 | 模板要求 | 指定机器持续 10 分钟不崩溃 |
| 服务端 | 成功率 ≥99.99%、P995 <1000ms、分档全局 QPS | 单房间聚合 max 1000、全局 4200、P995 <1000ms |
| 大场景 | 分主播规模 QPS | 对战单场支持 5 万参与 UV |

这些数字显然不是一套可直接拼接的验收标准。项目做法：

1. 两套都保留为压力设计参考。
2. 提测前向接入方书面确认当前客户端机器、FPS、全局 QPS、成功率和延迟基线。
3. 报告同时写“测试环境、并发模型、事件比例、持续时间、采样方法和原始监控图”，只写一个结果数字不算完成。

## 七、审核、发布与线上变更

### 7.1 提审材料

- 提案、基础信息和权属资料
- 正式 EXE 包、`ks_manifest.json`、规则素材
- 功能、客户端性能、服务端性能报告
- PC 开播与评论/点赞/送礼录屏
- 回调地址、scope、礼物/关键词配置
- 版本说明、已知问题、监控与故障联系人

公开页面没有给出统一的提案/包体审核 SLA，不应自行承诺日期。

### 7.2 技术总览当前给出的上线纪律

- 可监控、可灰度、可回滚，核心流程可限流和降级。
- 灰度示例：1% → 10% → 30% → 60% → 100%。
- 上线窗口：周一至周四 10:00–12:00、14:00–18:00；周五尽量不上，周末禁止。
- 新业务放量期间按要求巡检并在对接群通报。
- 无法快速止损时平台可能下线；修复后重新测试和放量。
- 禁止用热更新绕过审核。

这些更像已建联项目的协作纪律，实际排期以当期对接群为准。[技术接口总览](https://open.kuaishou.com/miniPlayDocs/technologyrelated/Affiliatedability/expand/Subsidiaryability)

### 7.3 最小线上监控

- 在线主播/房间数、启动成功率、异常掉量
- bind 状态、token 刷新、scope/鉴权失败
- callback 请求量、验签失败、非 200、`result != 1` 和重试
- 各 `pushType` 数量、延迟、重复率、`stress_` 隔离量
- 礼物收到 → 幂等消费 → 客户端展示的完整链路
- EXE 崩溃、FPS、内存、伴侣 IPC 与捕获失败
- 单房间和全局 QPS、队列积压、限流与降级状态

平台服务协议还要求，发现可能影响运营的问题时，24 小时内向平台提供书面说明、处理方案和时间计划。[小玩法平台服务协议](https://open.kuaishou.com/docs/operate/platformAgreement/miniplay/agreement)

## 八、商业化、结算与数据合规

### 8.1 商业化与保证金

- 平台可向部分玩法提供礼物和商业化能力，开发者与主播按平台规则获得分成。
- 公开文档没有统一固定分成比例，以后台或书面通知为准。
- 营收玩法可能需要缴纳保证金；金额和方式以后台为准。
- 保证金不足可能影响结算、服务或营收能力。

官方依据：[小玩法平台服务协议](https://open.kuaishou.com/docs/operate/platformAgreement/miniplay/agreement)、[保证金管理规范](https://open.kuaishou.com/docs/operate/platformAgreement/miniplay/epayServiceAgreement)。

### 8.2 当前公开结算流程

- 在后台账户管理维护银行、发票、税率和收件信息。
- 当前 SOP 写每月 2 日可获取上月结算单。
- 平台运营确认后由开发者确认数据与银行信息；开发者确认后不可修改。
- 电子发票当前支持 OFD/PDF，可一单多票。
- 审核通过后进入付款；驳回时按原因重传。
- 2024 年 7 月及之后的弹幕、派对、快乐玩法支持线上结算；更早周期为线下结算。

日期、票种、地址和账户信息均属于动态后台数据，不在知识库固化。[新结算平台操作流程](https://open.kuaishou.com/miniPlayDocs/Settlementpreparation/refundGuidelines)

### 8.3 数据与隐私

- 未经平台同意，不得擅自收集、存储、抓取或处理终端用户数据。
- 确需处理时，要有隐私政策、明确告知与同意，并遵循最少必要。
- 不得将平台数据用于玩法服务之外的营销、广告或第三方披露。
- 提供用户修改、删除数据的路径；停止服务后永久删除平台数据及备份。
- 对用户评论、昵称等 UGC 做合规处置，建立投诉机制并公示有效联系方式。
- 发生数据泄露时，按协议通知用户、平台与监管部门。

不要把普通快手小程序的数据分析、挂载或结算文档直接套到 PC 弹幕玩法。

### 8.4 下线与终止

- 主动终止服务应先与平台确认并取得所需书面同意。
- 提前通知用户、关闭新增付费入口，处理未到期权益、返还或补偿。
- 完成平台数据和备份删除，并留存不含个人敏感数据的执行记录。
- 不能以“包体不再维护”为由直接停服。

依据：[小玩法平台服务协议](https://open.kuaishou.com/docs/operate/platformAgreement/miniplay/agreement)。

## 九、常见故障定位

| 现象 | 优先检查 |
|---|---|
| 启动后要求切换其他玩法 | `workFileNames` 是否包含实际运行进程 |
| 手机观看模糊 | 观看清晰度、玩法区域分辨率是否达到要求 |
| 已拿到 roomCode 又收到 `SC_SET_CODE` | 以新 roomCode 更新绑定，不忽略 |
| 自动捕获失败 | `captureParam`、窗口名/类、进程名和捕获类型 |
| 一直显示“连接中” | 工作进程、IPC、开发者服务和快手服务四段链路 |
| 倒计时提示玩法不可用 | 是否手动重复添加玩法源、画面占比是否满足要求 |
| 回调重复或结果错乱 | 原始 body 验签、HTTP 200/`result=1`、幂等与乱序 |
| 正式榜单出现压测用户 | `stress_` 前缀和测试事件隔离是否在业务入口执行 |
| 权限/房间错误 | token、scope、AppID、roomCode、主播开播状态和频率 |

官方依据：[PC 客户端接入与 FAQ](https://open.kuaishou.com/miniPlayDocs/technologyrelated/Affiliatedability/basicability/clientside)、[启动失败案例](https://open.kuaishou.com/miniPlayDocs/technologyrelated/commonProblem/startupFailed)。

## 十二、发布闸门

### 商务与提案前

- [ ] 已区分互动小玩法/游戏小玩法及是否含内购。
- [ ] 主体、注册资本、软著、版号/备案由商务书面确认。
- [ ] 玩法不是普通游戏移植，主播能持续讲解，免费参与成立。
- [ ] 安全区、规则、礼物效果、结果和主播露脸位置已出图。
- [ ] 题材、术语、长期成长、商店、抽奖和榜单机制过最新准入页。

### 开发完成

- [ ] `AppID.exe`、`ks_manifest.json`、工作进程、清理进程配置正确。
- [ ] `-c/-pos/-ipc`、`SC_SET_CODE`、`SC_QUIT` 和断连均已验证。
- [ ] token 提前刷新、bind/stop、round、gift 和 scope 可观测。
- [ ] 原始 body 验签，`unique_message_id` 与 `uniqueNo` 双层幂等。
- [ ] 乱序、重复、断网、重启和 `stress_` 隔离测试通过。
- [ ] 未使用热更新或远程内容绕开审核。

### 提审与上线前

- [ ] 体验者、直播伴侣版本、包体替换方式已按当期后台确认。
- [ ] 功能、客户端、服务端报告采用平台确认的同一套基线。
- [ ] 开播、评论、点赞、每种礼物和退出录屏齐全。
- [ ] 审核包与上线包哈希一致，版本说明和已知问题完整。
- [ ] 灰度、监控、降级、回滚、巡检和紧急联系人就绪。
- [ ] 分成、保证金、结算、隐私和数据删除只按当前协议/后台执行。
