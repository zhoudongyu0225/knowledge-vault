# 抖音 × 快手弹幕小玩法：双平台差异与共用工程清单

## 先给结论

最小、可靠的双平台边界是：

> 共享玩法核心、渲染、状态机、标准事件、可靠性和测试回放；抖音与快手各自维护启动/房间、鉴权/签名、事件接入、包体、配置、测试和发布适配层。

不能共用的不是“几个接口地址”，而是完整的平台生命周期。不要把实现写成一个巨大的 `if (platform)`；也不需要一开始拆成很多微服务。对小团队，单仓库、一个玩法服务、两个平台适配模块、两套构建与发布配置已经足够。

## 一、统一架构

~~~mermaid
flowchart TB
    subgraph Platform[\"平台侧\"]
        DY[\"抖音伴侣 / 云启动<br/>token、数据任务、SDK、ACK\"]
        KS[\"快手 PC 伴侣<br/>roomCode、IPC、bind、callback\"]
    end

    subgraph Adapter[\"必须分开的平台适配层\"]
        DYA[\"DouyinAdapter<br/>鉴权 / 事件 / 启停 / ACK / 包体\"]
        KSA[\"KuaishouAdapter<br/>鉴权 / IPC / bind / 回调 / manifest\"]
    end

    subgraph Shared[\"可共用\"]
        ING[\"EventIngest<br/>验签后标准化、测试隔离、幂等、乱序\"]
        CORE[\"GameplayCore<br/>对局状态机、单位、技能、积分、榜单\"]
        CLIENT[\"ClientCore<br/>渲染、动画、音效、UI 组件\"]
        OBS[\"Observability<br/>日志、指标、告警、回放、降级开关\"]
    end

    DY --> DYA
    KS --> KSA
    DYA --> ING
    KSA --> ING
    ING --> CORE
    CORE --> CLIENT
    DYA --> CLIENT
    KSA --> CLIENT
    ING --> OBS
    CORE --> OBS
    CLIENT --> OBS
~~~

关键纪律：

- `GameplayCore` 不引用平台 SDK、AppID、roomCode、room_id 或平台用户类。
- 平台原始事件先验签，再进入共享标准化层。
- 礼物和用户 ID 只在各自平台命名空间内使用。
- 客户端“收到指令”和“完成展示”分别记录。
- 审核包、平台配置、文案和 UI 安全区不共用最终产物。

## 二、生命周期差异

| 环节 | 抖音 | 快手 | 共用资产 |
|---|---|---|---|
| 准入起点 | 开放平台注册/认证、创建 AppID、提案 | 商务提案与 PRD 在前，之后主体入驻与后台创建 | 玩法一页纸、互动循环、权属清单 |
| 主体/资质 | 个人或企业；当前直播玩法要求相应软著和授权链 | 当前最新类目主要按企业主体、互动/游戏小玩法区分资质 | IP、音乐、美术、字体授权管理 |
| 主播启动 | 伴侣 EXE 或获批云启动，传临时 token | PC 伴侣启动独立 EXE，传 `-c/-pos/-ipc` | 客户端生命周期状态机 |
| 房间建立 | token 换房间信息，启动数据任务 | roomCode 上报服务端，调用 bind `start` | 内部 session/room 抽象 |
| 服务端鉴权 | 相关接口按各自定义签名；直播回调另有验签链 | access_token + query 参数；回调原始 body + AppSecret MD5 | 密钥管理、刷新、告警框架 |
| 事件 | 礼物、评论、点赞、粉丝团为主 | 礼物、评论、点赞、关注、分享、粉丝团、贵族等 | 标准事件模型、业务路由 |
| 可靠性 | 重复、乱序、评论/点赞可能丢；有限回查；履约 ACK | 乱序、平台重试、`unique_message_id`/`uniqueNo`；`stress_` 隔离 | 幂等、乱序、队列、审计和回放 |
| 包体 | ZIP 结构、英文 EXE、版本、体积要求 | `AppID.exe + ks_manifest.json`、IPC、捕获、进程清理 | 共享资源构建，平台产物独立 |
| 测试 | OpenAPI mock、开播/用户调试成员、私密直播 | 自测包、体验者、伴侣、本地完整包替换、三类报告 | 用例、事件回放、性能工具 |
| 发布 | 定向/全量；当前定向转全量不可逆；迭代重审 | 包体重审、灰度、上线窗口与巡检；禁止热更新 | 版本说明、监控和应急预案 |
| 数据/结算 | 数据期限和服务费以当期规则/后台为准 | 协议严格限制数据处理；分成、保证金、结算看后台 | 数据最少化、删除、财务留档 |

## 三、什么可以共用

### 3.1 产品与内容

- 玩法规则、胜负条件、对局节奏和数值模型
- 评论/点赞/礼物对应的“意图”，例如加入阵营、加速、召唤
- 主播话术框架和观众新手引导的信息结构
- 美术源文件、动画、音效和通用 UI 组件
- 题材权属、素材来源和授权链台账

最终平台文案和布局仍要分开：抖音有直播玩法术语和内容规范，快手有自己的准入题材与 880×960 安全区。

### 3.2 客户端

- Unity/C# 玩法核心和渲染
- 单位、技能、动画、音频、对象池和数值
- 与平台无关的对局状态机
- 标准事件驱动的表现层
- 崩溃、FPS、内存和展示结果埋点框架
- 自动化回放与截图/录屏场景

### 3.3 服务端

- 标准事件总线
- 幂等键存储与去重窗口
- 按房间/对局路由
- 排行榜和结算算法
- 队列、背压、限流、降级和熔断
- 日志、指标、告警和事件回放
- 配置中心的结构

## 四、什么必须分开

| 模块 | 抖音专属 | 快手专属 |
|---|---|---|
| 启动参数 | `-token`、云启动/移动输入 | `-c`、`-pos`、`-ipc` |
| 房间生命周期 | token 换房间、数据任务启停 | roomCode、bind/round、`SC_SET_CODE`/`SC_QUIT` |
| SDK/回调 | Unity SDK、直推、履约 ACK | HTTP callback、伴侣 IPC |
| 鉴权/签名 | 各接口按当前定义鉴权；直播回调验签独立 | access_token、query、原始 body MD5 |
| 测试标识 | `test=true` | `LIVE_INTERACTION_DATA_TEST`、`stress_` |
| 礼物与评论 | 抖音后台礼物池、指令匹配 | 快手订阅礼物、评论精准匹配 |
| 包体 | ZIP、英文 EXE 与版本结构 | `AppID.exe`、`ks_manifest.json`、捕获/进程字段 |
| 提审发布 | 基础信息、版本提审、定向/全量 | 商务/PRD、三类报告、灰度与上线窗口 |
| 运营规则 | 抖音准入词表、合作与服务费 | 快手题材、安全区、保证金与结算 SOP |

## 五、标准事件模型

建议进入玩法核心前统一成：

~~~yaml
platform: douyin | kuaishou
app_id: string
room_id: string
round_id: string | null
event_id: string
event_type: GIFT | COMMENT | LIKE | FANS_JOIN | FOLLOW | SHARE | OTHER
user_id: string
occurred_at: timestamp | null
received_at: timestamp
is_test: boolean
payload:
  command: string | null
  gift_id: string | null
  gift_count: number | null
  amount_hint: number | null
raw_version: string
raw_event: restricted_object
~~~

设计约束：

- `user_id` 实际是 `platform + platform_user_id`，不跨平台合并身份。
- `event_id` 必须来自平台唯一标识；平台没有稳定 ID 时才生成组合键并标低置信度。
- `occurred_at` 缺失时不伪造，只保留 `received_at`。
- `raw_event` 只用于短期审计和故障回放，遵循平台数据最少化及删除要求。
- `amount_hint` 不作为财务结算真值；平台账单和礼物原始标识才是依据。
- 测试标识在业务入口拦截，不等排行榜写完后再清理。

### 5.1 事件映射

| 标准事件 | 抖音 | 快手 |
|---|---|---|
| GIFT | `live_gift` | `giftSend` |
| COMMENT | `live_comment` | `liveComment` |
| LIKE | `live_like` | `liveLike` |
| FANS_JOIN | `live_fansclub`（权限/页面需自测） | `fansGroupJoin` |
| FOLLOW | 专用直播玩法链路未明确 | `follow`（权限为准） |
| SHARE | 未作为当前主链路列出 | `share`（权限为准） |
| OTHER | 当前 AppID 获批扩展事件 | `noble`、超粉团等获批事件 |

产品逻辑只能依赖“两平台都已确认”的事件。平台独有事件通过可选能力接口进入，缺失时玩法仍能正常完成一局。

## 六、可靠性处理

### 6.1 消费顺序

~~~text
HTTP/SDK/IPC 接入
→ 平台验签或会话校验
→ 测试消息识别与隔离
→ 原始事件受限审计
→ event_id / gift unique id 幂等
→ 字段标准化
→ room + round 路由
→ 玩法状态更新
→ 客户端表现
→ 平台 ACK / 展示结果埋点
~~~

### 6.2 业务规则

- 评论和点赞按“提示/累计贡献”设计，不把单条必达作为胜负前提。
- 礼物按高价值通道处理：更长去重窗口、更高队列优先级、更强告警。
- 对局结束后迟到事件按 `round_id` 丢弃、补记或进入明确的迟到策略，绝不悄悄加到下一局。
- 重复事件可重复展示与否是产品选择，但积分、资产和收入只能结算一次。
- 平台回调成功、服务端消费成功、客户端收到、客户端展示是四个不同状态。
- 降级时先聚合点赞/评论表现，再保护礼物和房间生命周期。
- 任何平台密钥只放密钥管理/环境配置，不进客户端、不进 Git、不进知识库。

## 八、小团队协作

不要求每个角色对应一个人，但责任必须明确：

| 责任 | 主要产物 | 做完标准 |
|---|---|---|
| 产品/合规 | 玩法一页纸、规则、互动映射、红线检查 | 两平台提案差异可追踪，关键口径有官方来源 |
| 客户端 | 共享表现层、两套启动/SDK/IPC 适配、平台包体 | 伴侣启动、互动展示、退出和崩溃恢复通过 |
| 服务端 | 鉴权、事件接入、幂等、对局、监控 | 重复/乱序/丢失/重启/峰值测试通过 |
| 测试/发布 | 用例、录屏、报告、构建哈希、提审材料 | 审核包等于发布包，动态规则已复核 |
| 运营/主播 | 开播教程、话术、巡检、反馈收集 | 主播能独立开播，异常能按 runbook 止损 |

每次版本只指定一位“发布负责人”，负责冻结包体、确认平台配置、执行检查表和记录审核反馈，避免朋友之间出现“大家都以为别人确认过”。

## 九、测试矩阵

### 9.1 共享功能测试

- [ ] 一局开始、进行、结束、榜单、再来一局完整。
- [ ] 每个评论指令命中、未命中和高频重复。
- [ ] 点赞单次、连续、聚合和限流降级。
- [ ] 每一种礼物单个、连击、批量、重复和迟到。
- [ ] 免费用户可以参与并获得明确反馈。
- [ ] 付费反馈与规则一致，不出现随机零反馈。
- [ ] 主播画面、评论区、规则和关键 UI 均可读。

### 9.2 事件可靠性测试

| 故障注入 | 预期 |
|---|---|
| 相同事件重复 10 次 | 积分/资产/收入只记一次 |
| 评论、点赞、礼物乱序 | 状态不倒退，结果仍可解释 |
| 评论/点赞丢失 | 一局可继续，不发生永久等待 |
| 礼物处理后响应丢失 | 平台重试也不重复结算 |
| 事件跨局迟到 | 不进入下一局 |
| 回调超时/非 2xx | 告警、重试、积压可观测 |
| 服务端处理后崩溃 | 重启消费仍幂等 |
| 客户端掉线/重启 | 会话重建，不重放不可重复资产 |
| 测试事件 | 不进正式榜单、收入、资产和分析 |

### 9.3 平台专项测试

抖音：

- [ ] `-token`、房间信息和数据任务启停。
- [ ] `test=true` 隔离。
- [ ] 服务端回调、SDK/直推和履约 ACK。
- [ ] 取消挂载、关播、定向用户进入和私密直播。
- [ ] ZIP 结构、版本、依赖文件和体积。

快手：

- [ ] `-c/-pos/-ipc`、`SC_SET_CODE`、`SC_QUIT`、断连。
- [ ] bind `start/stop/status`、round 和礼物配置。
- [ ] `unique_message_id`、礼物 `uniqueNo`、`stress_` 隔离。
- [ ] `ks_manifest.json`、工作/清理进程和自动捕获。
- [ ] 功能/客户端/服务端报告采用当期确认基线。

## 十一、双平台发布闸门

### Gate 0：立项

- [ ] 玩法在真人直播中有持续互动价值，不是普通游戏移植。
- [ ] 免费参与路径、礼物反馈和一局闭环清楚。
- [ ] 两平台题材、术语、随机、付费和长期成长规则都通过预审。
- [ ] 权属链和所需主体资质已列清；不确定项已问平台。

### Gate 1：技术方案

- [ ] `Core` 不依赖平台 SDK 或平台用户标识。
- [ ] 两个平台适配层、配置和构建产物明确分开。
- [ ] 事件模型、幂等键、round 路由、测试隔离和 ACK/展示状态已设计。
- [ ] 密钥管理、数据最少化、删除期限和日志范围明确。

### Gate 2：可提测

- [ ] 正常互动和异常矩阵自动/人工验证通过。
- [ ] 私密直播完成真实端到端测试。
- [ ] 性能环境、流量模型、持续时间和原始监控图齐全。
- [ ] 主播能按教程独立启动、退出并处理常见故障。

### Gate 3：可提审

- [ ] 两套包体、版本、配置、截图、文案和礼物/关键词分别冻结。
- [ ] 审核包与计划上线包哈希一致。
- [ ] 所有动态规则在本次版本重新核对并记录日期。
- [ ] 自查表、录屏、性能报告、版本说明和授权资料齐全。

### Gate 4：可上线

- [ ] 开放范围/灰度计划和不可逆选择已确认。
- [ ] 房间、事件、礼物、客户端展示、ACK、崩溃和队列监控上线。
- [ ] 限流、降级、止损、回滚/重新发版方案和 on-call 明确。
- [ ] 主播、运营、开发知道故障时先做什么。
- [ ] 分成、保证金、结算和数据规则以当前后台/协议复核。

### Gate 5：可继续放量

- [ ] 启动/挂载、对局完成、互动处理和展示成功率稳定。
- [ ] 礼物重复结算为零，测试消息污染为零。
- [ ] 崩溃、延迟、掉房和积压在确认阈值内。
- [ ] 主播反馈、用户投诉和审核反馈有负责人处理。
- [ ] 新版本不依赖热更新绕审，能重新测试和提审。

## 十二、动态规则台账

这些数字不能进入共享业务常量：

| 类型 | 示例 | 应放在哪里 |
|---|---|---|
| 账号/测试配额 | 开播成员、调试用户、体验者 | 平台发布配置与版本记录 |
| 互动配置 | 礼物数量、评论词数量、点赞频控 | 平台后台配置快照 |
| 凭证/接口 | token 有效期、QPS、超时、重试、回查 | 平台 adapter 配置 |
| 客户端 | SDK/伴侣版本、包体大小、FPS | 构建与测试基线 |
| 服务端 | 单房间/全局 QPS、P995、成功率 | 容量计划 |
| 发布 | 审核时效、上线窗口、灰度比例 | 发布计划 |
| 商业 | 服务费、分成、保证金、礼物池 | 财务/协议记录 |
| 合规 | 题材、术语、数据期限、付费规则 | 合规检查表 |

## 十三、推荐开发顺序

为减少返工，首个双平台项目按这个顺序：

1. 用同一份玩法一页纸分别过两平台提案与红线，不急着做完整美术。
2. 先实现不依赖平台事件的玩法核心和本地事件回放。
3. 先选一个已有准入/对接更顺的平台跑通端到端发布链路。
4. 把已验证的核心保持不动，只新增第二个平台 adapter、配置、包体和测试。
5. 最后再做扩展事件、云开播、底部栏等白名单能力。

“先选一平台打通”只是执行顺序，代码边界从第一天就按双平台设计。
