# 个人知识库 Codex 协作规则

## 启动时必须读取

- 每次在本仓库开始新对话或新任务，先检查当前分支、`git status --short` 和远端跟踪状态；安全时先 `git fetch`，若上游有更新且工作区干净，则用 `git pull --ff-only` 同步后再读取上下文。
- 如果工作区有未提交改动、分支发生分叉、拉取会冲突或远端不可访问，停止自动同步并明确报告，不覆盖、不暂存、不改写任何已有内容。
- 每次在本仓库开始新任务，先读取 `_索引/Codex协作经验复盘与行为准则.md`。
- 继续既有工作或进行项目决策时，再读取 `_索引/当前工作状态.md` 及其中关联的产品档案、数据分析或专题文档。
- 文档与用户当前消息冲突时，以用户当前消息为准，并明确指出冲突。

## 工作方式

- 先定义问题、口径、完成标准和证据强度，再分析或修改文件。
- 核心业务结论必须包含数据、来源、判断、缺失信息和下一步动作，避免只写抽象描述。
- 正式归档保留最新、最准确、最可执行版本；重复内容合并，不为一次性任务增加复杂结构。
- 只有重大结论、当前优先级或未完成事项发生变化时，才更新 `_索引/当前工作状态.md`。

## 公众号关注池

- 用户明确新增关注公众号时，当次更新 `40_资料库/游戏发行与广告/公众号专题/持续关注公众号池.md`，不等待周度任务。
- 周度公众号检查只遍历池中状态为 `active` 的账号；同一账号的别名和跨平台入口合并维护。

## 敏感信息与 Git

- 内部数据、付费资料、聊天记录和原始截图默认私有。
- 不得提交 token、密码、cookie、认证文件或 `90_附件/私有资料/` 下的原始敏感数据。
- Git 同步前检查差异；无改动不创建空提交，发生冲突时停止并报告，不强制覆盖。
